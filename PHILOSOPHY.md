# 일용할묵상 — Product Philosophy

> 말씀을 중심으로 나의 습관을 훈련하고, 같은 말씀을 읽은 사람들과 조용히 동행하며,
> 시간이 지나며 내가 어떤 사람으로 형성되고 있는지 발견하게 하는 플랫폼.

이 문서는 두 AI(Claude, Codex)와 향후 모든 기여자가 공유하는 제품 정체성입니다.
새 기능을 검토할 때 항상 이 기준에 비춰 결정합니다.

---

## 1. Product Identity

### 우리가 만드는 것
**영성 형성 플랫폼** — 말씀이 사람의 삶을 다듬는 운영체제.

### 우리가 만들지 않는 것
- 생산성 앱
- 습관 트래커
- 묵상 SNS
- 큐티 자료실

### 영적 형성의 흐름
```
말씀 → 묵상 → 한 줄 말씀 → 기도 → 실천 → 삶의 훈련 → 기억 → 형성
```

---

## 2. Formation Principles

### 묵상은 중심이다
모든 다른 기능은 묵상에서 흘러나오거나 묵상으로 돌아간다.
- 챌린지는 묵상의 동반 훈련
- 공유는 묵상의 공명
- Streak은 묵상의 흔적

묵상이 여러 체크박스 중 하나처럼 보이면 안 된다.

### 한 줄 말씀은 영적 시그널이다
사용자가 매일 받는 한 문장이 형성의 핵심 데이터.
- 모든 묵상 흐름의 끝에 위치
- 가장 시각적으로 강조 (Noto Serif KR, 큰 사이즈)
- 시간이 지나면 본인이 어떤 말씀을 받았는지 발견 가능

### 시간은 KST다
사용자가 살아내는 "오늘"은 한국 시간 자정 ~ 자정.
- DB 함수 `public.kst_date()` + 클라이언트 `getTodayDateString()` 동일 기준
- streak, 잔디, 알림 시간 모두 KST

### 실패는 정죄가 아니라 회복이다
Duolingo의 "❌ Streak lost!"는 우리 톤이 아니다.
- 끊긴 후: "오늘 다시 이어갈 수 있어요"
- 빠뜨린 날: 빈 칸 (X 표시 X)
- 빨간색은 경고용으로만 (드물게)

## 3. UX Guardrails

### 절대 피할 것
- 좋아요 경쟁 / 글로벌 랭킹
- 팔로우 중심 SNS 구조
- 댓글 난장판
- 챌린지 앱처럼 보이는 홈
- 묵상이 체크리스트 한 줄로 격하되는 구조
- 알림으로 사용자를 추격
- ERP/CMS/교회 행정 인상

### 항상 지향할 것
- 묵상은 항상 가장 위, 가장 크게
- 챌린지는 "함께 걷는 훈련" 톤으로 한 단계 아래
- 공유는 조용한 동행 ("오늘 함께 머문 말씀")
- 숫자는 동기부여용이지 정체성의 중심이 아님
- 큰 여백, 부드러운 모션, 세리프 인용

### 모든 기능 판단의 기준
> "더 많이 기록하게 하는가?"가 아니라 **"더 깊이 형성하는가?"**

## 4. Community & Sharing Principles

### 사람이 다리가 아니라 말씀이 다리다
일반 SNS는 팔로우/친구 그래프로 사람을 연결한다.
이 앱은 **같은 본문을 읽은 사람들**을 자동으로 연결한다.

### 공유 안전장치 (Phase 9 설계 시 필수)
1. **본인 먼저**: 자기 묵상 작성 후에만 다른 사람 묵상 볼 수 있음
2. **opt-in 기본 private**: 사용자가 명시적으로 "공유" 선택해야 함
3. **익명 옵션**: 닉네임 / 익명 둘 다 가능
4. **댓글 없음**: 공감 스티커만 (🙏 🌱 ❤️ 등)
5. **글로벌 랭킹 없음**: 좋아요 수는 본인에게만 보임 (있더라도)
6. **24시간 쿨다운** (선택): 작성 직후 노출 X — 즉각 보상 차단
7. **신고/모더레이션**: 임계치 넘으면 자동 임시 숨김 → 검토

### 큐레이션 > 알고리즘
"인기 묵상" 대신 **"오늘 함께 머문 말씀"**.
운영자/목회자 추천 묵상이 알고리즘 추천보다 우선.

### Cold start 대응
Phase 9 런칭 시점에 Phase 10 큐레이션도 같이 활성화.
빈 광장이 사용자의 마지막 화면이 되지 않도록.

## 5. Identity Policy

### 식별 방식
- **닉네임 기본**
- 실명 미사용
- 익명 공유 선택 가능

### 정책 요약

| 항목 | 결정 |
|---|---|
| 기본 식별 | 닉네임 (`display_name` + `handle`) |
| 실명 | 사용하지 않음 |
| 익명 공유 | 가능, **영구 익명** (one-way door) |
| 익명 공유 글 → 닉네임 전환 | 불가 |
| 닉네임 변경 | 가능, **월 1회** (`handle_changed_at`로 cooldown) |
| 닉네임 공유 글 | 현재 닉네임 따라감 |
| 아바타 | seed 기반 procedural (Boring Avatars 권장) |
| 이미지 업로드 | 초기 없음 |
| Handle 충돌 | 자동 제안 + 직접 입력 |
| 예약어 | `admin`, `support`, `운영`, `하나님`, `예수` 등 차단 |
| 부적절 닉네임 | 단어 필터 + 신고 기반 처리 |
| 계정 삭제 | 사용자 선택: 모두 삭제 / 익명화 보존 |

### 기존 사용자 마이그레이션
- 자동 기본 닉네임 부여: `벗-{6자리}` (예: `벗-a3k7p2`)
- 첫 공유 시도 시 닉네임 변경 onboarding
- 닉네임 미설정 상태에선 공유 불가 (읽기/묵상/저장은 가능)

### author_snapshot
`shared_reflections.author_snapshot jsonb`는 선택적으로 보관.
- 닉네임 공유 글: 표시는 현재 `profiles.display_name` 우선
- snapshot은 정책 변경 또는 계정 삭제 시 fallback

## 6. AI Collaboration Rules

이 프로젝트는 두 AI(Claude Code, ChatGPT Codex)와 인간이 함께 개발한다.

### 공통 규칙
- 모든 AI는 이 문서를 항상 읽고 시작
- 새 기능 추가 시 **Section 3 (UX Guardrails)** 의 기준에 비춰 자체 검토
- 묵상 중심 원칙(Section 2)을 깨는 변경 제안 금지
- 데이터 모델 변경은 **하위 호환** (nullable 컬럼, optional props)
- 시간은 KST 통일 (Section 2)
- 실패 UX는 정죄 톤이 아니라 회복 톤 (Section 2)

### Claude / Codex 동기화
- 작업은 GitHub `main` 브랜치 기준
- 한 쪽이 마이그레이션 파일을 만들면 다른 쪽은 동일 timestamp 충돌 피하기
- 큰 변경은 PR로, 작은 fix는 직접 commit
- 마이그레이션 파일명은 `YYYYMMDDHHMMSS_short_name.sql` (timestamp 정렬)

### 새 기능 검토 체크리스트
- [ ] 묵상을 격하시키지 않는가?
- [ ] 더 깊이 형성하는가? (단순히 더 많이 기록하게 하는 게 아니라)
- [ ] 실패 톤이 회복적인가?
- [ ] 시간대 KST 일관인가?
- [ ] 닉네임/익명 정책과 충돌하지 않는가?
- [ ] 하위 호환되는가?

## 7. 구현 로드맵 (참고용)

```
Phase 1~6  완료 ──  영성 시그널, 홈 입구화, 여정화, Notion화, 디자인 폴리시
Phase 7    완료 ──  함께 걷는 훈련 (개인 챌린지, reflection_id 연결)
Phase 8    완료 ──  훈련 회고 (잔디, 회복 UX, 형성 기록, 말씀 안에서)
Phase 9    함께 시작하기 (챌린지 카테고리 공유)
             A 베이스 + B 따뜻한 한 조각:
             - 카테고리별 인기 챌린지 집계 (랭킹 X, 단순 시작 수)
             - "이번 주 새로 시작한 분들" (격려용)
             - "나도 시작" 복사 흐름 (copied_from)
             - 비교/경쟁 노출은 절대 안 함
Phase 10   같은 본문 묵상 나눔 (닉네임/익명, 본인 먼저)
Phase 11   좋은 묵상 발견 (큐레이션, Phase 10과 동시 활성)
```

챌린지 공유를 묵상 공유보다 앞에 둔 이유:
챌린지는 개인 훈련이라 사회적 비교 위험이 묵상보다 낮음.
공유 UX 패턴을 챌린지에서 검증한 후 묵상으로 확장.

각 phase는 이 문서의 원칙을 준수해야 한다.
원칙과 충돌하는 phase 계획은 phase가 아니라 원칙을 의심한다.
